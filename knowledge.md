## Core Philosophy
Reqly is an execution engine, not an AI product. The AI always lives outside Reqly. Its job is to expose reliable, well-typed tools that any AI agent can call.

- **Tool-First:** If it can't be called via MCP, it doesn't exist.
- **Dumb Server:** The server fires HTTP requests and manages files. No LLM logic.
- **Plain Text:** Collections are YAML. No binary formats, no proprietary DB.
- **BYOK:** The user's API key stays on their machine. Nothing sent to Reqly servers.

## Competitive Advantages (vs Postman / Insomnia / Bruno)

| Advantage | Detail |
|-----------|--------|
| **Git-native YAML collections** | Every competitor stores collections in a proprietary format (Insomnia: NeDB binary `.db` files; Postman: cloud-locked). Reqly's `.reqly/` YAML is human-readable, git-diffable, and directly writable by AI agents from the codebase. |
| **MCP server, not MCP client** | Insomnia recently added an MCP *client* (can call external tools). Reqly *is* the MCP server - AI agents connect once and get the full toolset with no UI required. Fundamentally different architecture. |
| **Auto-capture, three ways** | Outbound proxy (`reqly exec npm run dev`), inbound middleware (`reqly-middleware`), and public webhook tunnel. No other tool covers all three capture modes. |
| **BYOK + no cloud** | No Reqly cloud. No sync subscription. Collections travel with the repo via git. The prompt bar forwards to the user's own API key. |
| **Response diffing** | Detects what changed between runs - status, latency, body. No other mainstream API client does this. |

## Architecture
- **Runtime:** Node.js + TypeScript
- **Server:** Express/Fastify serving MCP stdio interface and localhost web UI.
- **UI:** React, served as static build.
- **Config:** `~/.reqly/config.json` for global settings (reserved for future preferences - no BYOK key, agents talk to Reqly via MCP).

## What's Built
- **Core Engine:** HTTP executor (with >50KB response truncation), Collection Management (YAML), Environment substitution, Auth profiles.
- **MCP Server:** Tools to create, list, run requests/collections, manage environments and variables (create/get/set/delete), retrieve full untruncated responses, and manage proxy.
- **Local UI:** Left icon navigation rail (Collections / Environments / History / Capture / Settings) driving switchable sidebar panels, Request Editor with params/headers/body/auth/assertions, Response Viewer with syntax highlighting and assertions results, Settings panel (placeholder for future preferences), Multiple Tabs support for parallel editing. Variable autocomplete triggers on `{{` in all text inputs and textareas to quickly inject environment variables.
- **Differentiators:** 
  - **Auto-Capture Proxy:** Captures outbound dev traffic and saves requests.
  - **Public Webhook Tunnel:** Exposes `localhost` via `localtunnel` to catch incoming webhooks from external services (Stripe, etc.) and save them directly into your collections.
  - **Test Assertions:** Verify response status, latency, and JSON body paths.
  - **Collection Runner:** Sequentially run all requests in a collection with pass/fail tracking.
  - **Request Chaining:** Downstream requests can access previous response data via `{{requestName.response.path}}`.
- **Environment Editor:** Full CRUD for environments and variables from the UI nav rail - create, rename-by-recreate, edit variables inline, delete with confirmation. Backed by `POST/PUT/DELETE /api/environments`.
- **Collection Manager UI:** Full CRUD from sidebar - right-click context menus on collections (Add Request, Rename, Delete) and requests (Rename, Duplicate, Delete), inline rename via Enter/Escape, +New collection input. Backed by existing `/api/collections` CRUD endpoints.
- **Request History:** In-memory log (last 200) of every fired request - timestamp, method, URL, status, latency. History panel in nav rail lists newest first, click to load request into editor, Clear button. Backed by `GET/DELETE /api/history`. Appended on adhoc runs, MCP `run_request`, and collection runs.
- **Search / Command Palette:** ⌘K / Ctrl+K overlay searches across collection names, request names, and URLs. Keyboard-navigable (arrow keys, Enter, Escape). Results grouped by type, click opens request in editor. Runs over in-memory `GET /api/collections` - no backend search index. Trigger button in the top bar is a wide (`w-72`) pill sitting left-of-center, not crammed against Settings.
- **Request Tabs:** Closeable X buttons, unsaved-changes dot indicator (per-tab saved snapshot diff), active-tab blue underline, left/right scroll arrows for overflow. Live edits flow back from RequestEditor via `onChange` for dirty tracking.
- **Variables Tab:** Read-only tab in the request editor showing the active environment's variables (Key as `{{name}}`, Value, source = active env name). Helps debug unresolved variables. Editing stays in the Environments panel.
- **Collection Variables:** Per-collection variables stored in a reserved `collection.yaml` metadata file inside each collection folder. Always available to every request in that collection regardless of the active environment, and they win over environment variables of the same name. Resolution uses a layered scope chain (`resolveVariables(template, [collectionVars, envVars])`, first layer wins) that also coexists with `{{request.response.path}}` chaining. MCP tools `get_collection_variables`/`set_collection_variable`/`delete_collection_variable` and REST routes `GET/PUT/DELETE /api/collections/:name/variables[/:key]`. UI: "Settings" item in the collection right-click menu opens `CollectionSettingsModal` (Variables tab, draft + explicit Save, same pattern as the Environments panel); the request editor's Variables tab shows a Key/Value/Source table merging collection and env vars, collection vars listed first since they win on collision.
- **Collection Auth (engine + MCP):** Optional `auth` field in `collection.yaml` metadata. Auth precedence at execution time: request-level auth (including explicit `type: none`) > collection auth > nothing. An explicit `type: none` on a request suppresses collection auth entirely. `resolveCollectionAuth` in `src/engine/collection-auth.ts` resolves a `CollectionAuth` config into a concrete `AuthProfile` (profileId lookup with inline-credential fallback). `http-executor.execute` accepts `collectionAuth` as a lowest-precedence fallback. Threaded through `run_request` MCP tool, `CollectionRunner`, and the `/api/run/adhoc` Express route. MCP tools: `get_collection_auth`/`set_collection_auth`/`delete_collection_auth`. REST routes: `GET/PUT/DELETE /api/collections/:name/auth`. UI in T-091.
- **GraphQL Workspace:** Dedicated nav rail icon (between History and Capture) opens a full-area GraphQL workspace (`src/ui/src/components/GraphQLWorkspace.tsx`), replacing the REST tab bar/editor entirely rather than living inside it. URL bar, Query editor (CodeMirror, schema-aware highlighting once introspected) + Variables sub-tab (JSON), Introspect button, Send button, and a Response panel (reuses `ResponseViewer`). State is local to the workspace - GraphQL requests are not REST collection items and do not appear in the Collections sidebar or get saved to YAML. On Send it assembles `{ query, variables }` as the request body via the same `/api/run/adhoc` endpoint REST requests use - the HTTP executor stringifies object bodies and sets `Content-Type: application/json` transparently, so no executor or MCP changes were needed. The REST `RequestEditor` no longer has any GraphQL mode or toggle.
- **Pre/Post-run Scripts:** Per-request JavaScript scripts that execute in a sandboxed `vm.runInNewContext` context (Node built-in, no external deps, 2s timeout). `preScript` runs before substitution - env mutations flow into the URL/header/body interpolation. `postScript` runs after the response - exposes `response.body`, `response.status`, etc. for extracting tokens or values into `env`. Errors are isolated (caught and logged). UI: "Pre-script" and "Post-script" tabs in the request editor with monospace textarea. Engine: `src/engine/script-runner.ts`.
- **GraphQL Support:** First-class GraphQL in the engine. `RequestConfig` carries `type: 'graphql'` and a `graphql: { query, variables? }` field. The HTTP executor auto-builds the POST body and Content-Type. GraphQL requests are stored as YAML in `.reqly/` alongside REST requests and run transparently via `run_request`. UI GraphQL workspace (syntax-highlighted editor, schema introspection, variables panel) has a Save button that persists to any collection.
- **Response Diffing:** After each `run_request` (MCP or UI), the response viewer shows a "Diff" tab when changes are detected vs the previous run. Status changes, latency delta, and body diffs (added/removed JSON keys, or line diff for non-JSON). Engine: `src/engine/response-differ.ts`. History entries now store up to 10 KB of body for comparison.
- **Import from Postman/Bruno:** `reqly import <file>` CLI and `import_collection` MCP tool accept a Postman v2.1 JSON file or a Bruno `.bru` file/directory. Parses and writes requests directly into `.reqly/` YAML collections. UI sidebar has an Upload icon that accepts `.json`/`.bru` files via browser file picker. Engine: `src/engine/importer.ts`.
- **Visual Polish:** Shared `lib/colors.ts` helpers enforce a consistent Hoppscotch palette across all panels - method badges (GET=#22c55e, POST=#eab308, PUT=#3b82f6, PATCH=#f97316, DELETE=#ef4444, rendered as pills via `methodBadgeClass`/`METHOD_BADGE_BASE`) and status badges (2xx green, 3xx blue, 4xx yellow, 5xx red). Replaced per-component inline color switches in CollectionsPanel, HistoryPanel, SpotlightSearch, CapturePanel, App tab bar, and ResponseViewer. All icons across the app (nav rail, tab close/new, send/save, tree expand arrows, environment edit/delete, key-value enable/remove, response spinner) use Lucide React (`lucide-react`) instead of hand-rolled inline SVG. Nav rail active state is a colored chip (`bg-blue-500/10` rounded square) plus left accent bar, not just a color swap.
- **UI State Persistence:** Open tabs, active tab, and active nav panel persist across page refreshes via a debounced `useLocalStorage` hook (`src/ui/src/hooks/useLocalStorage.ts`). Tabs are sanitized before writing - response bodies and auth credentials are stripped (ephemeral / sensitive). Active environment already persists server-side in `environments.yaml` (`store.active`), so no config.json change was needed.
- **CLI Runner & Distribution:** Added `reqly run <collection>` for headless execution with JSON and TAP reporters for CI/CD integration. The engine is fully installable globally via `npm install -g @rutvikpansare123/reqly` and features a `reqly setup` command to natively configure MCP servers in Cursor and Claude without manual JSON editing.
- **Project Resolution:** The server resolves its `.reqly` project root via a priority chain - `--project-dir` flag > `REQLY_PROJECT_DIR` env var > `activeProject` field in `~/.reqly/config.json` (set via `reqly use <path>`) > `process.cwd()`. `reqly use` exists for MCP hosts like Claude Desktop that spawn one global server process with no per-project launch context. `reqly status` prints which source won and what path resolved, for debugging.
- **`reqly exec` / `exec_with_proxy`:** Eliminates the manual `HTTP_PROXY=...` step for traffic capture. `reqly exec npm run dev` starts the auto-capture proxy, runs the command with `HTTP_PROXY`/`HTTPS_PROXY` injected and `stdio: 'inherit'` (output looks identical to running the command directly), forwards Ctrl+C to the child, and prints a capture summary on exit. The MCP tool `exec_with_proxy` does the agent-driven equivalent - it spawns the command as a detached background process (logging to `~/.reqly/exec.log`) so the agent doesn't block waiting on a long-running dev server, and only asks the user to run the command manually (returning `fallbackCommand`) if spawning fails. `stop_proxy` kills the tracked child process group as well as stopping the proxy, so one call cleans up both.
- **Developer Tools:** cURL import (paste any cURL command, fields auto-populate); code snippet generation (fetch/axios/curl, one-click copy); TypeScript interface generator (infers typed interface from any JSON response body, client-side, no server round-trip).
- **Collection Export:** Export any collection as Postman v2.1 JSON or OpenAPI 3.0 JSON via right-click context menu in the sidebar, `GET /api/collections/:name/export`, or the `export_collection` MCP tool.
- **Script Console:** `console.log/warn/error` calls in pre/post scripts are captured with level prefixes and shown in a dedicated Console tab in the response viewer (log=gray, warn=amber, error=red). MCP `run_request` responses include `consoleLogs` automatically.
- **Inbound Capture Middleware (`reqly-middleware`):** `reqlyMiddleware()` (Express), `reqlyMiddlewareHook()` (Fastify), `reqlyNextMiddleware()` (Next.js, via `reqly-middleware/next`). One line added to the app captures every inbound request (fire-and-forget POST to `localhost:4242/capture/inbound`, errors swallowed, never blocks the app) into a named collection, same dedup format as the outbound proxy. The `install_middleware` MCP tool detects the project's framework from `package.json` and hands the agent the exact install command + snippet so it can wire this up proactively without the user asking. Local dev only by default.
- **Workflow Guidance for Agents:** Every MCP tool `description` states what it does, when to use it, and (where relevant) the preferred call sequence, so an agent connecting for the first time reads the intended workflow directly from the tool list rather than guessing. A `reqly://workflow` MCP resource gives a short plain-text ranked guide: PRIMARY (read the codebase, build the collection from route files) vs SECONDARY (proxy capture for outbound third-party traffic).
- **Single-Instance Enforcement:** Only one Reqly process holds port 4242 at a time, tracked via a lock file at `~/.reqly/running.json` (`pid`, `projectDir`, `port`, `startedAt` - see `src/server/lock.ts`). When `reqly start --project-dir X` is run while another instance is already alive, the new process calls `POST /api/switch-project` on the running instance (hot-swaps its `collectionManager`/`environmentManager` to point at `X`, updates the lock) instead of trying to bind the port itself - it then starts only its own MCP stdio server (no Express, "mcp-only mode") so each AI tool's stdio session stays correctly project-scoped. Stale locks (dead pid) are detected via `isProcessAlive` and cleared automatically. `reqly stop` reads the lock and posts `POST /api/shutdown` to gracefully stop the running instance.
