Capabilities

One engine. Thirty-plus tools.
Your agent drives.

Reqly is not an AI. It is an execution engine your AI connects to via MCP. Connect once, and your agent gets the full toolkit.

Build collections

Agent reads your routes and scaffolds requests automatically. No YAML by hand, no clicking through forms.

Fire and assert

Runs requests, checks status, body, and latency. Returns pass/fail per assertion in a structured shape agents can parse.

Chain flows

Multi-step e2e sequences with extract, poll, and conditional branching. Chains REST, GraphQL, and gRPC in the same flow.

Validate contracts

Point a collection at an OpenAPI spec. Every response is checked against it automatically on every run.

Serve mocks

Fake a backend from saved examples. Frontend dev and agent testing without a live server.

Capture traffic

Proxy for outbound, one-line middleware for inbound, public tunnel for webhooks. Collections build themselves.

Export anywhere

Generate OpenAPI specs, Postman files, or Markdown API docs. You are never locked in.

Real auth support

Bearer, API Key, Basic, OAuth 2.0 with PKCE, mTLS, and AWS SigV4 signing for API Gateway, Bedrock, AppSync, and S3.

Every protocol

REST, GraphQL with schema introspection, gRPC with server reflection, WebSocket, SSE, Socket.IO, and MQTT.

Architecture

The zero-human pipeline

One prompt. Tests running
in CI forever.

No manual Actions YAML. No environment wiring. Your agent chains Reqly's MCP tools end to end in a single session.

STEP 01

Agent builds the collection

"Read my Express routes and build a collection with assertions." The agent calls create_collection and create_request for every endpoint it finds in your code.

STEP 02

Agent writes the e2e flow

"Write a flow for the login → checkout path." The agent calls create_flow and add_flow_step, runs it, and iterates until every step is green.

STEP 03

Agent ships it to CI

"Export it to CI." One call to export_flow_ci writes a GitHub Actions workflow that installs Reqly, runs the flow, and uploads JUnit results.

Git-native by design

Collections are plain YAML.
In your repo. Diffable.

No proprietary formats, no cloud account, no binary database. Collections live in .reqly/ and travel with your code via git. Your git provider is your RBAC.

name: create-user
method: POST
url: "{{baseUrl}}/api/users"
headers:
  Content-Type: application/json
body:
  email: test@example.com
assertions:
  - field: status
    operator: eq
    value: 201
  - field: body
    path: user.id
    operator: neq
    value: ""
steps:
  - type: run
    request: auth/login

  - type: extract
    from: response.body.token
    into: authToken   # now {{authToken}}

  - type: run
    request: posts/create-post

  - type: assert
    field: status
    operator: eq
    value: 201
# written by export_flow_ci - you never touch this
name: e2e-post
on: [push, pull_request]
jobs:
  flow:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install -g getreqly
      - run: reqly run-flow "e2e-post" --reporter junit > results.xml
      - uses: actions/upload-artifact@v4
        with:
          path: results.xml
