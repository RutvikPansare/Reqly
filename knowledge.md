These apply on every task, no exceptions. Full detail in CLAUDE.md / GEMINI.md.

A feature with no MCP tool is invisible to the agent ecosystem. ## Core Philosophy
Reqly is an execution engine, not an AI product. The AI always lives outside Reqly. Its job is to expose reliable, well-typed tools that any AI agent can call.

- **Tool-First:** If it can't be called via MCP, it doesn't exist.
- **Dumb Server:** The server fires HTTP requests and manages files. No LLM logic.
- **Plain Text:** Collections are YAML. No binary formats, no proprietary DB.

## Protocol Architecture Rule: Request/Response vs. Long-Lived Connections

This distinction is **mandatory reading before implementing any new protocol**.

### Request/Response protocols (REST, GraphQL, gRPC)
The server owns the full lifecycle. Engine class executes the request, returns a result. MCP tool and Express route both call the same engine. UI calls the Express route and renders the result. This is the standard Reqly pattern.

```
Agent / UI → MCP tool or POST /api/run/adhoc
           → Engine (grpc-runner, http-executor, etc.)
           → External server
           ← Result returned to caller
```

### Realtime protocols (WebSocket, SSE, Socket.IO, MQTT)
Connections are long-lived - potentially open for minutes or hours. A server-side proxy (holding the connection on the Reqly Node.js process and SSE-piping messages to the browser) creates serious problems: double-hop latency on every message, sessions lost on server restart, doubled file descriptors per session, and a fundamental mismatch with the singleton-server model.

**The correct split:**

**For agents (MCP use case):** A buffered executor - connect, capture messages for `captureTimeout` seconds, disconnect, return `{ messages, truncated }`. Same pattern as `grpc-streaming.ts`. Stateless and ephemeral. Lives in `src/engine/realtime-executor.ts`, exposed via MCP `run_realtime` tool and Express `/api/run/realtime` (for the UI to trigger ad-hoc captures).

**For humans (interactive UI use case):** Direct browser connections using native and browser-compatible APIs. No proxy hop. Sessions survive server restarts.
- WebSocket: `new WebSocket(url)` - native browser API, zero packages
- SSE: `new EventSource(url)` - native browser API, zero packages
- Socket.IO: `socket.io-client` in `src/ui/package.json` (browser-compatible build)
- MQTT: `mqtt` in `src/ui/package.json` (browser-compatible build via WebSocket transport)

**What this means in practice:**
- `ws`, `eventsource` (Node.js versions) are NOT added to root `package.json` for a proxy. They are not needed.
- `socket.io-client` and `mqtt` ARE added to `src/ui/package.json` for the browser panels.
- The MCP `run_realtime` tool covers the "verify this endpoint works" agent use case via a short-lived buffered capture.
- The interactive UI provides a richer live session experience that agents don't replicate directly - and that is acceptable because the two use cases are fundamentally different.
- Realtime requests ARE saved to collections as YAML (`type: websocket`, `type: sse`, `type: socketio`, `type: mqtt`) so agents can reference them and re-run them via `run_realtime`.

## Competitive Advantages (vs Postman / Insomnia / Bruno)

| Advantage | Detail |
|-----------|--------|
| **Git-native YAML collections** | Every competitor stores collections in a proprietary format (Insomnia: NeDB binary `.db` files; Postman: cloud-locked). Reqly's `.reqly/` YAML is human-readable, git-diffable, and directly writable by AI agents from the codebase. |
| **MCP server, not MCP client** | Insomnia recently added an MCP *client* (can call external tools). Reqly *is* the MCP server - AI agents connect once and get the full toolset with no UI required. Fundamentally different architecture. |
| **Auto-capture, three ways** | Outbound proxy (`reqly exec npm run dev`), inbound middleware (`reqly-middleware`), and public webhook tunnel. No other tool covers all three capture modes. |
| **No cloud** | No Reqly cloud. No sync subscription. Collections travel with the repo via git. |
| **Response diffing** | Detects what changed between runs - status, latency, body. No other mainstream API client does this. |

## Architecture
- **Runtime:** Node.js + TypeScript
- **Server:** Express/Fastify serving MCP stdio interface and localhost web UI.
- **UI:** React, served as static build.
- **Config:** `~/.reqly/config.json` for global settings (reserved for future preferences - no BYOK key, agents talk to Reqly via MCP).

## What's Built
- **Core Engine:** HTTP executor (with >50KB response truncation), Collection Management (YAML), Environment substitution, Auth profiles.
- **Secret Providers (T-245 + T-249):** `SecretProviderRegistry` in `src/engine/secret-providers/` maps vault URI prefixes (`op://`, `vault://`, `aws://`, `bw://`) to providers; a known prefix with no registered provider fails loudly. Providers: Bitwarden Secrets Manager (`bw://project-name/secret-name` via `@bitwarden/sdk-napi`, token from `BITWARDENSM_ACCESS_TOKEN` + `BITWARDENSM_ORGANIZATION_ID` or `secretProviders.bitwarden.{accessToken,organizationId}`) 1Password (`op://vault/item/field` via `@1password/sdk`, token from `OP_SERVICE_ACCOUNT_TOKEN` or `secretProviders.onepassword.serviceAccountToken`), AWS Secrets Manager (`aws://secret-name` or full ARN via `@aws-sdk/client-secrets-manager`, standard AWS credential chain, no credential storage in Reqly), and HashiCorp Vault (`vault://mount/data/path/field`, direct KV v2 HTTP with `VAULT_ADDR` + `VAULT_TOKEN` or `secretProviders.vault.{address,token}`), config in `~/.reqly/config.json`. Vault URIs placed as values in `.env` resolve at load time into the variable chain; failed keys are excluded from execution and any request referencing one fails with the provider's error. Inline `{{secret:<vault-uri>}}` works in any request field. `reqly secrets resolve` writes plaintext values to `.env.local`. MCP tools: `get_secret` (4-char preview), `get_secret_status` (URI list + state, no values), `configure_secret_provider` (persist config + live re-resolve). - **Single-Instance Enforcement:** Only one Reqly process holds port 4242 at a time, tracked via a lock file at `~/.reqly/running.json` (`pid`, `projectDir`, `port`, `startedAt` - see `src/server/lock.ts`). When `reqly start --project-dir X` is run while another instance is already alive, the new process calls `POST /api/switch-project` on the running instance (hot-swaps its `collectionManager`/`environmentManager` to point at `X`, updates the lock) instead of trying to bind the port itself - it then starts only its own MCP stdio server (no Express, "mcp-only mode") so each AI tool's stdio session stays correctly project-scoped. Stale locks (dead pid) are detected via `isProcessAlive` and cleared automatically. `reqly stop` reads the lock and posts `POST /api/shutdown` to gracefully stop the running instance. Agents have MCP parity for project switching: `get_project` returns the current project directory, `switch_project` does the same hot-swap as `POST /api/switch-project` directly against the in-process context (no HTTP round-trip). For humans, the sidebar's project path widget is an interactive switcher: click it to expand into a folder-icon (calls `GET /api/open-folder-picker` for a native OS folder dialog) plus a text input. Submitting checks the target path exists (inline "Path not found" error if not) and that it has a `.reqly/` folder (a modal offers to create one if missing, or to pick a different folder). A soft, non-blocking warning appears if an AI agent made an MCP tool call in the last 60 seconds (`lastMcpActivityAt` on `GET /api/project`), since switching changes that agent's project context too.
- **VS Code Extension (T-240/241/242):** `packages/vscode/` ships the `reqly.reqly` extension for VS Code, Cursor, Windsurf, and all VS Code-compatible editors - a thin client over the local server's REST API at `localhost:4242` (configurable via the `reqly.serverUrl` setting), no engine logic duplicated. Activity-bar collection tree (projects -> collections -> requests; clicking a request previews its on-disk YAML read-only; inline run buttons), status bar `Reqly: <env>` environment switcher, command palette commands (Run Request / Run Collection / Switch Environment / Start Proxy / Open UI), a "▶ Run with Reqly" CodeLens above fetch/axios/got calls in JS/TS (runs the matching saved request, or offers "Save as new request"), and YAML schema validation for all `.reqly/` files (schemas generated from `src/types/` at build time via `ts-json-schema-generator`, contributed through redhat.vscode-yaml - typo squiggles, enum autocomplete, hover docs). Published to the Marketplace by `.github/workflows/vsce-publish.yml` on `v*` tags.
- **Named Workspaces (T-226):** `~/.reqly/workspaces/<name>/workspace.yaml` maps stable repo aliases to machine-local paths (`repos: [{ alias, path }]`, optional `sharedEnv`) - the foundation for cross-repo flows (T-227). Machine-local, never committed; alias names are stable across teammates while each developer links their own paths. Engine `src/engine/workspace-manager.ts`; active workspace persisted as `activeWorkspace` in `~/.reqly/config.json`. CLI `reqly workspace create/link/use` (plus the pre-existing `add/remove/list` for the flat multi-project list); MCP `create_workspace`, `link_workspace_repo`, `use_workspace`, `list_workspaces`; REST under `/api/workspaces`; UI workspace dropdown above the project list with a settings modal for linking/unlinking repos.
- **Mock Server:** `reqly mock <collection>` (default port 4243) serves a collection's saved example responses as a real HTTP server - route table built from `MockPathResolver` (`{{var}}` URL segments become `:var` Express params; an explicit `mockPath` on a request always wins) for every request that has at least one saved example. CORS is permissive by default. `X-Reqly-Example` header picks a named example, falling back to the first. 404s list the available routes. Agent parity via `start_mock`/`stop_mock`/`get_mock_status` MCP tools and `/api/mock/*` Express routes; UI: a "Mock" tab in the Capture panel (collection picker, port, start/stop, live route table, click-through to the request).
- **.env File Integration:** Zero-config - if `.env` exists at the project root it's loaded automatically, no setup needed. Resolver priority is collection vars > active env vars > `.env` (lowest). Multiple files supported (`['.env', '.env.local']`, later wins on collision), persisted in `~/.reqly/config.json`, hot-reloads on file change with no restart. CLI: repeatable `--env-file <path>` overrides for one session. Agents manage the file list via `set_dotenv_files`/`get_dotenv_files` MCP tools (values never returned over MCP/REST - use `get_variables` to resolve one at runtime); `get_variables` now tags every result with its `source`. UI: Settings panel has an "Environment files" list (add/remove/reorder); the request editor's Variables tab and autocomplete show dotenv keys with the filename as source, value always masked.
- **OpenAPI Contract Validation:** A collection can be pointed at an OpenAPI 3.0/Swagger 2.0 spec (local file path or URL, configured via the Collection Settings modal's "Contract" tab or the `set_collection_spec` MCP tool) - every response fired against that collection is then matched to a spec operation (by `specOperationId` on the request, or by inferring the path/method) and validated: undefined status codes and body-schema mismatches are errors, Content-Type mismatches are warnings. Surfaces everywhere a response shows up: `run_request` and `POST /api/run/adhoc` both return `contractViolations`, the response viewer has a "Contract" tab (green "all checks passed" / red violation list with a count badge / amber "no matching operation" with the inferred path), `validate_response` re-checks a stored response without re-firing, and `reqly run <collection> --validate-spec` fails CI on any violation. Agents have full parity via `set_collection_spec`/`get_collection_spec`/`delete_collection_spec`/`list_spec_operations`/`validate_response` MCP tools.
- **Multipart Body (engine + MCP + UI):** Requests can send `multipart/form-data` by setting `body: { type: 'multipart', parts: [...] }`. Text parts carry their value inline; file parts carry a `filePath` relative to the project root (file contents never stored in YAML). The executor builds a `FormData`, lets undici/fetch set the `Content-Type: multipart/form-data; boundary=...` header automatically, and returns a structured `{ error: "File not found: ..." }` without throwing if a file is missing. The `create_request` MCP tool schema exposes the full multipart body shape to agents. The UI adds a "Multipart" body type to the request editor with a part table (Name, Text/File type toggle, value input or file picker, Content-Type override via a `···` toggle, hover-reveal trash), an "Add part" button, and an inline notice about file paths when file parts are present. Ad-hoc runs with browser-picked files use a dedicated `POST /api/run/adhoc/multipart` route (multer memStorage) that passes file buffers to the executor via a `resolvedFiles` map.
- **Flow CI Export:** `reqly export-flow <name> --format github-actions` (or the `export_flow_ci` MCP tool) writes `.github/workflows/<name>.yml` - a GitHub Actions workflow that installs Reqly, starts it, runs the flow with `--reporter junit > results.xml`, and uploads `results.xml` via `actions/upload-artifact@v4`, with a commented-out "Start server" step the developer can uncomment for flows that hit a local API. Pure string templating (`generateGithubActionsWorkflow`), no YAML library. The Express route variant (`POST /api/flows/:name/export-ci`) deliberately does not write to disk - it returns the YAML as a string for the UI to copy/download.
- **Embedded Terminal:** A "Terminal" nav rail icon opens a real shell in the localhost UI - `child_process.spawn` over a `/terminal` WebSocket (`src/server/terminal.ts`), one command at a time per connection, cwd always the current project root (re-read live, so it follows `/api/switch-project`). Output renders via `@xterm/xterm` for ANSI support; commands are entered through a separate React input bar (not xterm's own stdin) with up/down arrow history (last 50). Kill button sends SIGTERM then SIGKILL after 2s; a killed process reports a non-zero exit code with the signal name rather than falsely showing a clean exit. No auth - matches Reqly's existing localhost-only trust model (`exec_with_proxy` already spawns processes the same way). Human-facing only, no MCP tool - agents already have `exec_with_proxy` and the CLI.
- **Move Request Between Collections:** Drag a request onto a different collection's header (blue highlight while a valid drop target, no-op onto its own collection), or right-click it and choose "Move to..." for a modal-based picker - both call the same `moveRequest` move, which appends ` (1)`, ` (2)` etc. if the target already has a request with that name. The moved request auto-opens in its new collection afterward. MCP parity via `move_request`; REST via `POST /api/collections/:collection/requests/:request/move`.
- **Duplicate Collection/Environment:** Right-click a collection ("Duplicate" in the existing menu) or an environment (new right-click menu, previously had none) to deep-copy it as `"Copy of <name>"`, incrementing to `(1)`, `(2)` etc. on collision. Collection duplication copies the whole folder (all requests + metadata); environment duplication copies all variables. MCP parity via `duplicate_collection`/`duplicate_environment`; REST via `POST /api/collections/:name/clone` (named "clone" to avoid colliding with the pre-existing per-request `/duplicate` route) and `POST /api/environments/duplicate`.
- **Agent Onboarding Nudge:** When a project has zero collections, the main canvas replaces the request editor with `EmptyStateNudge` - copyable prompts ("Build from routes", "Import from OpenAPI spec", "Write an e2e flow") to paste into any MCP-connected agent, framework-aware via `detectFramework`/`GET /api/project`. A one-time setup banner (`reqly setup <cursor|claude|claudecode|gemini>`) shows until `hasEverConnectedAgent` flips true. The sidebar shows a matching one-line nudge when its collection list is empty. Disappears permanently (via a localStorage flag) the first time a collection is created, even if later deleted.
- **Starter Collection (`reqly init`):** `reqly init` copies a bundled example collection (`example/reqly-starter/.reqly/`, against the free JSONPlaceholder API - no auth, no setup) into the project's `.reqly/`, merging without overwriting existing files. Demonstrates collection variables, a `postScript` that extracts a response value into an env var, request chaining via `{{requestName.response.body.field}}`, and a flow. The `reqly://getting-started` MCP resource points agents at these same patterns on first connect.
- **Test Flows:** A flow is an ordered sequence of steps (`run`/`extract`/`assert`/`poll`/`conditional`) stored as YAML in `.reqly/flows/<name>.yaml`, separate from collections - SoapUI-style e2e automation on top of saved requests. `FlowRunner` (`src/engine/flow-runner.ts`) executes steps with a flow-local variable scope layered above collection vars and env vars, supports retry on `run` steps, `goto`/`skip`/`abort` branching on `conditional` (circular-goto detection), polling with an `until` expression (no arbitrary JS eval - a small safe comparison/truthiness evaluator), and data-driven runs (once per row in `flow.data`, or a single ad-hoc `dataRow` override). Agents have full parity via 8 MCP tools (`create_flow`, `get_flow`, `list_flows`, `delete_flow`, `add_flow_step`, `update_flow_step`, `delete_flow_step`, `run_flow`) and matching `/api/flows` REST routes. `reqly run-flow <name>` (`src/server/run-flow-command.ts`) runs a flow headlessly for CI - `pretty`/`json`/`tap`/`junit` reporters, `--data-row '<json>'` for ad-hoc single-row runs, exit code 1 on any failed step. UI: a dedicated "Flows" nav rail icon opens a sidebar flow list (`FlowsPanel`) and a main workspace (`FlowWorkspace`) with a step list (`FlowStepCard` - status circle, type badge, expandable response detail), a right-side data-rows panel when the flow has data, a bottom results bar, an inline add-step picker for all 5 step types, and a Settings modal (`FlowSettingsModal`) for name/description/data rows.
